# 项目协作规则

## 项目定位

- 这是一个用于学习 Java 后端开发的技术社区项目，目标是通过可运行功能逐步达到企业项目开发和校招面试要求。
- 采用项目驱动、问题驱动和面试反向学习，不先完整学习 Java 理论再开始项目。
- 优先保证代码容易理解、功能可以验收，再考虑复杂抽象和性能优化。
- 每次只完成当前任务直接需要的改动，不提前生成整个系统，也不顺手重构无关代码。

## 固定技术栈

- 后端以 `backend-learning-planner.md` 为准：Java 17、Spring Boot 3、Maven、Spring MVC、MyBatis-Plus、MySQL、Redis、Spring Security、JWT、Swagger 和 Docker。
- 前端使用 Vue 3、TypeScript、Vite、Pinia、Vue Router、Axios 和 Element Plus。
- 未经用户明确要求，不替换核心框架；Elasticsearch、RabbitMQ、Linux 部署和微服务属于后续扩展，不进入当前十天主线。

## 规范开发流程

- 每个功能严格执行：明确目标与验收标准 -> 分析所需技术 -> 设计数据和接口 -> 说明文件计划 -> 实现 -> 测试 -> 总结原理 -> 整理面试表达。
- 每天按可独立验收的功能点拆分为多个代码生成批次；一次只允许执行一个功能点，不能在一次生成中顺带实现后续功能点。
- 开始每个代码批次前先说明设计、关键知识、验收标准和预计修改文件；避免一次生成大量代码。
- 每次生成或修改代码后都必须完成自检和相关测试，更新当天学习文档与开发总结，并给出针对该功能点的学习建议。
- 每个代码批次交付后都标记为“待用户 Review”并立即停止。即使属于同一天，用户确认前也不能执行下一个功能点的提示词。
- 用户确认当前功能点后，在当天学习文档和开发总结中记录 Review 结果，再进入下一个功能点；当天所有功能点确认后才允许执行 Git commit。
- 用户提出修改意见时继续处理当前任务，重新测试并再次请求确认。
- Review 重点包括企业规范、设计问题、安全与一致性、测试覆盖，以及面试中如何介绍该功能。

## 后端约束

- 按 `Controller -> Service -> Mapper -> Database` 分层；Controller 只处理 HTTP、参数校验和响应转换，业务逻辑放在 Service。
- API 使用 DTO 作为输入和输出，不直接暴露数据库 Entity/DO。
- 使用构造器注入；事务边界放在 Service；异常通过统一异常处理器转换为稳定的 JSON 响应。
- 使用 MyBatis-Plus 处理基础持久化；复杂查询应使用清晰、可测试的 Mapper SQL，不在 Service 中拼接 SQL。
- 数据库结构变化维护版本化 SQL 脚本，不依赖运行时自动修改表结构。
- 密码使用 BCrypt，密钥和数据库密码使用环境变量，不提交或输出敏感信息。
- 权限必须在后端校验。区分未认证的 401、无权限的 403，并校验文章、评论等资源的所有者。
- 点赞、收藏等接口必须考虑幂等性、数据库唯一约束和并发一致性。
- 接口使用 Swagger/OpenAPI 维护可调试文档；关键行为必须有日志、参数校验和统一错误码。

## 前端约束

- 使用 Vue Composition API、`<script setup>` 和 TypeScript。
- Pinia 只保存跨页面共享状态；HTTP 请求集中在 API 模块，不在页面中散落 Axios 配置。
- 页面必须处理加载、空数据、校验失败和请求失败状态；不能只实现成功路径。
- 前端权限控制只改善交互，不能替代后端鉴权。

## 代码质量与验证

- 遵循仓库已有命名和目录结构；修改前先阅读相关代码，不覆盖用户已有改动。
- 为新增业务规则补充相应测试，至少覆盖成功路径和一个关键失败路径。
- 完成改动后运行相关测试或构建；无法运行时明确说明原因，不能声称已经通过。
- 保持方法和类职责单一，避免为教学项目引入没有实际收益的通用框架或过度封装。
- 每日任务的项目开发、技术学习、原理理解和面试总结时间比例约为 60%、20%、15%、5%。

## 文档维护

- `docs/PROJECT_OVERVIEW.md` 是项目总概述；项目范围、架构、模块或技术决策变化时必须同步更新。
- `docs/DEVELOPMENT_SUMMARY.md` 是持续开发总结；每次添加或修改代码后追加记录，包括功能、文件、接口、数据变化、测试结果、学习建议和用户确认状态。
- `docs/learning/day-XX.md` 保存每天的详细学习任务、按功能点划分的代码生成提示词、完成记录和 Review 结果；当天实施过程中持续更新，不另建重复的临时计划。
- 代码、测试和对应文档视为同一个任务，文档未更新则任务未完成。

## 沟通方式

- 默认使用中文解释，代码标识符保持英文。
- 面向 Java 初学者说明关键调用链和设计原因，但不逐行复述显而易见的代码。
- 实现型任务结束时给出改动结果、验证方式、学习建议、当前限制和待确认事项。
