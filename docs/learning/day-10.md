## 今日开发任务

1. 确定文章详情、搜索和点赞 3 个压测场景及测试数据。
2. 在未额外优化前执行基线压测，记录并发数、吞吐量、P95 和错误率。
3. 结合日志、SQL 和 Redis 命中情况定位一个主要瓶颈。
4. 只针对证据实施优化，例如补充索引、修正查询、提高缓存命中或使用原子更新。
5. 对高频接口增加学习级限流，并返回明确的 429 响应。
6. 使用相同参数再次压测，输出前后对比和结论。
7. 创建后端 Dockerfile、前端 Dockerfile 和 Docker Compose，启动 MySQL、Redis、后端和前端。
8. 使用环境变量传入数据库、Redis 和 JWT 配置，不把密钥写进镜像。
9. 完善启动说明、接口地址、测试账号、架构和已知限制。
10. 更新项目总概述、开发总结和十天复盘，提交用户最终 Review。

## 原理学习

1. 为什么性能优化必须先有可重复的基线数据。
2. 平均响应时间为什么可能掩盖长尾问题，P95/P99 有什么价值。
3. 单机压测结果为什么不能等同于生产高并发能力。

## 面试问题

问题：如何定位一个接口的性能瓶颈？  
回答：先建立可复现压测，结合应用日志、慢 SQL、EXPLAIN、连接池、CPU、内存和缓存命中率逐层定位。  
追问：为什么不能看到接口慢就直接加 Redis？

问题：限流有哪些常见算法？  
回答：固定窗口、滑动窗口、漏桶和令牌桶，各自在突发流量、平滑程度和实现复杂度上不同。  
追问：固定窗口有什么临界突发问题？

问题：Docker Compose 在项目中解决什么问题？  
回答：声明并统一启动多个依赖服务、网络、端口和环境变量，提高本地环境可重复性。  
追问：Compose 是否适合大规模生产编排？

问题：如何介绍本项目的高并发优化？  
回答：说明基线指标、发现的具体瓶颈、实施的单项优化、相同参数下的对比结果和仍存在的边界。  
追问：下一阶段会如何继续优化？

## 验收标准

- 3 个核心接口完成基线压测，其中至少 1 个完成优化前后对比。
- 报告包含并发参数、吞吐量、P95、错误率和结论，不只展示截图。
- 限流触发时返回 429，正常请求不受明显影响。
- Docker Compose 能启动 MySQL、Redis、后端和前端。
