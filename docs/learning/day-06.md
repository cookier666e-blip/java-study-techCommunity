## 原理学习

1. 自关联表如何表示评论关系，为什么当前项目限制两级。
2. 什么是 N+1 查询，如何通过批量查询或联表减少 SQL 次数。
3. 评论删除后保留占位与直接删除对回复链的影响。

## 面试问题

问题：数据库如何设计多级评论？  
回答：常见方式有父子自关联、根评论加父评论、路径枚举和闭包表；本项目使用 root_id 与 parent_id 支撑两级结构。  
追问：如果支持无限层级会选择什么方案？

问题：什么是 N+1 查询？  
回答：先查询一批主记录，再为每条记录额外查询关联数据，导致查询次数随记录数增长。  
追问：如何发现和解决？

问题：评论应该物理删除还是逻辑删除？  
回答：取决于审计和上下文需求；社区通常保留记录或占位，避免破坏回复关系，同时隐藏内容。  
追问：逻辑删除会带来哪些查询风险？
